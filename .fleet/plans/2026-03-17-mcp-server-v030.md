# MCP Server Implementation (v0.3.0 P1)

## Summary

Expand the APEX MCP server from 6 tools to 20+, and add `apex integrate` for
zero-config setup in Claude Code, Cursor, and Windsurf. All MCP tools are thin
subprocess wrappers following the existing `mcp.rs` pattern.

## Current State

**Existing MCP tools (6):** `apex_run`, `apex_detect`, `apex_reach`,
`apex_ratchet`, `apex_doctor`, `apex_deploy_score`

**CLI subcommands not yet exposed (24):** Index, TestOptimize, TestPrioritize,
DeadCode, Lint, Diff, FlakyDetect, Complexity, Docs, AttackSurface,
VerifyBoundaries, RegressionCheck, Risk, Hotpaths, Contracts, Features,
SecretScan, LicenseScan, FlagHygiene, ApiDiff, DataFlow, BlastRadius,
ComplianceExport, ApiCoverage, ServiceMap, SchemaCheck, TestData

## Wave 1 ŌĆö `apex integrate` subcommand (no dependencies)

### Task 1.1 ŌĆö platform crew
**Add `Integrate` subcommand and `integrate.rs` module**

**Files:** `crates/apex-cli/src/lib.rs`, `crates/apex-cli/src/integrate.rs`

The `apex integrate` command auto-writes MCP server config for supported editors.
It detects the apex binary path and writes the appropriate JSON config.

Target editor config locations:
- **Claude Code:** `.mcp.json` in project root (or `~/.claude/settings.local.json` for global)
- **Cursor:** `.cursor/mcp.json` in project root
- **Windsurf:** `~/.codeium/windsurf/mcp_config.json`

```
apex integrate                    # auto-detect editor, write project-local config
apex integrate --editor claude    # explicit editor
apex integrate --editor cursor
apex integrate --editor windsurf
apex integrate --global           # write to user-global config instead of project
apex integrate --dry-run          # print config to stdout without writing
```

- [ ] Create `integrate.rs` module with config generation logic

- [ ] Write tests: `--dry-run` prints to stdout without file I/O

## Design Notes

### `apex integrate` ŌĆö Config Merging
When writing to `.mcp.json` or similar, the command must:
1. Read existing config if present
2. Merge the `apex` entry into `mcpServers` (do not clobber other servers)
3. Write back with pretty-printed JSON
4. Print confirmation message with the file path written

### MCP Tool Naming Convention
All MCP tools use `apex_` prefix with snake_case: `apex_complexity`,
`apex_dead_code`, `apex_test_optimize`, etc. This matches the existing 6 tools.

### Output Format
All new tools pass `--output-format json` to the subprocess when the subcommand
supports it. This gives structured output that AI assistants can parse.
Tools without JSON output (e.g., `features --output-format text`) still return
text content ŌĆö the MCP `Content::text()` wrapper handles both.

### Path Validation
All tools with `target`, `migration`, `schema`, `spec`, `old`, `new` path
parameters use `validate_target_path()` before passing to subprocess.

### Tool Descriptions
Each tool description should be 1-2 sentences explaining what data it returns
and when an AI assistant should call it. These descriptions are what LLMs see
when deciding which tool to invoke.
