## Wave 1 -- Adoption Blockers (P0, week 1)

No cross-crew dependencies. All tasks are independent.

### Task 1.1 -- platform crew: SARIF CLI output
**Files:** `crates/apex-cli/src/main.rs`, `crates/apex-cli/src/lib.rs`
**Effort:** 0.5 days
**Description:** Wire the existing `apex_detect::sarif::SarifReport` to the CLI via `--output-format sarif`. The SARIF module (`crates/apex-detect/src/sarif.rs`) already generates valid SARIF 2.1.0 from `Finding` structs. This task connects it to the CLI output path.
**Acceptance criteria:**
- [ ] `apex run --target <path> --output-format sarif` writes valid SARIF JSON to stdout
- [ ] `apex run --target <path> --output-format sarif -o findings.sarif` writes to file
- [ ] Output validates against SARIF 2.1.0 schema
- [ ] Existing `--output-format json` behavior unchanged
- [ ] Test: run against fixture project, parse output with `serde_json`, verify `$.runs[0].results` is non-empty

### Task 1.2 -- runtime crew: lcov export
**Files:** `crates/apex-instrument/src/lcov_export.rs` (new), `crates/apex-instrument/src/lib.rs`
**Effort:** 1 day
**Description:** Implement the inverse of `parse_lcov()` in `import.rs`. Convert APEX's internal `CoverageData` / `BranchId` representation to LCOV format for consumption by Coverage Gutters (VS Code), Codecov, and other tools.
**Acceptance criteria:**
- [ ] `pub fn export_lcov(data: &CoverageData) -> String` produces valid LCOV text
- [ ] Round-trip test: `parse_lcov(export_lcov(data)) == data` for representative inputs
- [ ] LCOV output includes `TN:`, `SF:`, `FN:`, `FNDA:`, `DA:`, `LF:`, `LH:`, `end_of_record` markers
- [ ] Branch coverage records (`BRDA:`) emitted when branch data is available
- [ ] Test: export a known coverage set, validate line counts match

### Task 1.3 -- platform crew: `--changed-files` filter
**Files:** `crates/apex-cli/src/main.rs`, `crates/apex-cli/src/lib.rs`
**Effort:** 1 day
**Description:** Add `--changed-files <base-ref>` CLI flag that limits analysis to files changed since `<base-ref>` (git diff). Runs `git diff --name-only <base-ref>...HEAD` internally and filters the file set before passing to the analysis pipeline.
**Acceptance criteria:**
- [ ] `apex run --target . --changed-files main` only analyzes files in `git diff --name-only main...HEAD`
- [ ] `apex run --target . --changed-files HEAD~3` works with commit refs
- [ ] Findings are filtered: only findings in changed files appear in output
- [ ] When no files changed, output is empty with exit code 0
- [ ] Error handling: graceful message when git is not available or ref is invalid
- [ ] Test: create a fixture with known diff, verify only expected files analyzed

### Task 1.4 -- platform crew: `apex badge` subcommand
**Files:** `crates/apex-cli/src/badge.rs` (new), `crates/apex-cli/src/main.rs`
**Effort:** 0.5 days
**Description:** Generate an SVG coverage badge (shields.io-compatible) from analysis results. Outputs a static SVG file that can be embedded in README.md.
**Acceptance criteria:**
- [ ] `apex badge --coverage 85.3 --output badge.svg` generates a valid SVG
- [ ] Color thresholds: red (<50%), orange (50-70%), yellow (70-80%), green (80-90%), brightgreen (>90%)

- [ ] Test: generate badge, verify SVG structure and color for each threshold

## Wave 3 -- Taint Engine & CPG Upgrades (P2, weeks 3-4)

Foundation work that unblocks Wave 4 detectors. Must complete before security-detect Wave 4 tasks.

### Task 3.1 -- security-detect crew: Tree-sitter CPG parser for Python
**Files:** `crates/apex-cpg/src/builder.rs`, `crates/apex-cpg/src/lib.rs`, `crates/apex-cpg/Cargo.toml`
**Effort:** 3 days
**Description:** Replace the line-based regex CPG builder with a tree-sitter-based builder for Python. The current builder misses nested calls, decorators, comprehensions, multi-line statements, and string interpolation. Implement `TreeSitterPythonCpgBuilder` implementing the existing `CpgBuilder` trait.
**Acceptance criteria:**
- [ ] Add `tree-sitter` and `tree-sitter-python` dependencies to `apex-cpg/Cargo.toml`
- [ ] `TreeSitterPythonCpgBuilder` implements `CpgBuilder` trait
- [ ] Correctly parses: nested calls (`foo(bar(x))`), decorators (`@app.route`), list comprehensions, multi-line statements, f-strings, method chaining
- [ ] All existing CPG tests pass with new builder
- [ ] New tests for each previously-broken construct (minimum 10 new tests)
- [ ] Existing `PythonCpgBuilder` retained as fallback behind a flag
- [ ] Benchmark: tree-sitter builder is no more than 2x slower than regex builder on same input

### Task 3.2 -- security-detect crew: Wire TaintSummary into taint engine
**Files:** `crates/apex-cpg/src/taint.rs`, `crates/apex-cpg/src/taint_summary.rs`
**Effort:** 1 day
**Description:** Connect the existing `TaintSummary` and `SummaryCache` to `find_taint_flows()`. When backward BFS reaches a `Call` node, look up the callee's summary. If the summary maps parameter N to return, continue BFS through the call. If no summary exists, overapproximate.
**Acceptance criteria:**
- [ ] `find_taint_flows()` accepts an optional `&SummaryCache` parameter
- [ ] When a Call node is reached, callee summary is consulted
- [ ] Summary-aware taint: `source -> foo(source) -> sink` is detected when `foo` has a summary showing param-to-return flow
- [ ] Overapproximation: unknown callees propagate taint through all args to return
- [ ] Test: multi-function taint flow that requires summary chaining (minimum 5 new tests)
- [ ] No regression on existing single-function taint tests

### Task 3.3 -- security-detect crew: HashMap graph storage upgrade
**Files:** `crates/apex-cpg/src/lib.rs`
**Effort:** 0.5 days
**Description:** Replace `Vec<(NodeId, NodeKind)>` with `HashMap<NodeId, NodeKind>` and build adjacency lists for edges. Makes `node()` O(1) and `edges_from()`/`edges_to()` O(degree) instead of O(n)/O(e).
**Acceptance criteria:**
- [ ] `Cpg::node()` is O(1) lookup
- [ ] `Cpg::edges_from()` and `edges_to()` use adjacency list, O(degree)
- [ ] All existing CPG tests pass without modification
- [ ] Benchmark: 10x improvement on graph with 10K+ nodes (add benchmark test)
- [ ] `Cpg::merge()` still works correctly with new storage

### Task 3.4 -- foundation crew: Compound Coverage Oracle
**Files:** `crates/apex-coverage/src/compound.rs` (new), `crates/apex-coverage/src/oracle.rs`, `crates/apex-coverage/src/lib.rs`
**Effort:** 2 days
**Description:** Implement the Bayesian log-odds coverage signal combiner from Dig 5. Multiple coverage signals (instrumented, imported, mutation, concolic, static, AI) are combined with confidence weights into a single coverage probability per branch.
**Acceptance criteria:**
- [ ] `CompoundOracle` struct combines multiple `CoverageSignal` values per `BranchId`
- [ ] Confidence tiers: Instrumented (1.0), Import (0.95), Mutation (0.85), Concolic (0.80), Static (0.50), AI (0.35)
- [ ] Log-odds combination: `combined = sigmoid(sum(log_odds(confidence_i * signal_i)))`
- [ ] Graceful degradation: works with any subset of signals (even just one)
- [ ] Coverage-informed severity: `adjusted_severity = base * (2.0 - coverage_confidence)`
- [ ] Test: combine 3 signals, verify combined confidence is between individual values
- [ ] Test: single signal produces that signal's confidence unchanged

### Task 3.5 -- runtime crew: WRAP mode CLI and injection engine
**Files:** `crates/apex-cli/src/wrap.rs` (new), `crates/apex-instrument/src/wrap.rs` (new), `crates/apex-cli/src/main.rs`, `crates/apex-instrument/src/lib.rs`
**Effort:** 3 days
**Description:** Implement `apex wrap -- <command>` that wraps user test commands with coverage injection. Supports Python (COVERAGE_PROCESS_START), JS/TS (NODE_V8_COVERAGE), Go (-coverprofile), Rust (RUSTFLAGS), Java (JAVA_TOOL_OPTIONS + JaCoCo). Reuses existing `import.rs` parsers for output.
**Acceptance criteria:**

- [ ] `apex wrap -- npm test` sets NODE_V8_COVERAGE and parses V8 coverage
- [ ] `apex wrap -- go test ./...` appends `-coverprofile` and parses Go coverage
- [ ] `apex wrap -- cargo test` sets RUSTFLAGS for instrument-coverage
- [ ] Language auto-detection from command name (pytest->Python, npm->JS, go->Go, cargo->Rust)
- [ ] `--lang <LANG>` override for ambiguous commands
- [ ] `--timeout <SECONDS>` with default 300s
- [ ] Coverage data fed into existing `CoverageOracle` pipeline
- [ ] Test: mock command execution for each language, verify coverage parsing

## Wave 5 -- Paradigm Shifts (P3, weeks 5-6)

Depends on Wave 3 (taint engine, compound oracle, CPG upgrades). These are the research-backed innovations.

### Task 5.1 -- intelligence crew: LLM concolic solver
**Files:** `crates/apex-synth/src/llm_solver.rs` (new), `crates/apex-synth/src/lib.rs`
**Effort:** 2 days
**Description:** Implement LLM as the third solver in the existing `PortfolioSolver` pipeline: gradient -> Z3 -> LLM. When gradient descent and Z3 both fail to solve a path constraint, serialize the `ConditionTree` to a human-readable source constraint and prompt an LLM to generate satisfying inputs. Based on ConcoLLMic (S&P 2026).
**Acceptance criteria:**
- [ ] `LlmSolver` implements the solver trait used by `PortfolioSolver`
- [ ] Accepts a `ConditionTree` and produces candidate input values
- [ ] Uses `ConditionTree::to_source_constraint()` for readable LLM prompts
- [ ] Configurable LLM endpoint (env var `APEX_LLM_ENDPOINT` or config file)
- [ ] Cost guard: only invoked after gradient + Z3 fail (estimated ~15 calls per project)
- [ ] Timeout: 10s per LLM call, 3 retries with backoff
- [ ] Test: mock LLM endpoint, verify solver produces valid inputs for known constraints
- [ ] Test: verify solver is NOT called when gradient or Z3 succeed

### Task 5.2 -- exploration crew: Wire LLM solver into PortfolioSolver
**Files:** `crates/apex-concolic/src/search.rs`, `crates/apex-concolic/src/lib.rs`
**Effort:** 1 day
**Description:** Register the `LlmSolver` from Task 5.1 as the third solver in the portfolio. The existing search strategy tries solvers in order; LLM becomes the last resort.
**Acceptance criteria:**
- [ ] `PortfolioSolver` includes `LlmSolver` when LLM endpoint is configured
- [ ] Solver order: gradient -> Z3 -> LLM (LLM only fires on gradient+Z3 failures)
- [ ] When no LLM endpoint configured, portfolio behaves as before (2 solvers)
- [ ] Statistics tracking: count of LLM invocations, success rate, cost estimate
- [ ] Test: end-to-end concolic run with mocked LLM, verify it fires only on hard branches

### Task 5.3 -- intelligence crew: Mutation-guided test generation
**Files:** `crates/apex-synth/src/mutation_gen.rs` (new), `crates/apex-synth/src/lib.rs`
**Effort:** 3 days
**Description:** Implement Meta ACH-style mutation-guided LLM test generation. For each function: (1) generate mutations (operator swap, boundary shift, return value change), (2) prompt LLM to generate a test that would catch the mutation, (3) validate the test kills the mutant. Based on Meta ACH (FSE 2025).
**Acceptance criteria:**
- [ ] `MutationTestGenerator` struct with `generate_tests(function: &Function) -> Vec<TestCase>`
- [ ] Mutation operators: arithmetic swap (+/-), comparison flip (< to >=), return value change, null injection
- [ ] LLM prompt includes: original function, mutated function, instruction to write a test distinguishing them
- [ ] Generated tests are syntactically valid (parse check before output)
- [ ] Cost control: batch mutations per function, max 5 LLM calls per function
- [ ] Test: mock LLM, verify mutation operators produce valid mutants
- [ ] Test: generated test structure matches expected format for Python and JS

### Task 5.4 -- security-detect crew: Type-state analysis
**Files:** `crates/apex-cpg/src/typestate.rs` (new), `crates/apex-cpg/src/lib.rs`, `crates/apex-detect/src/detectors/mod.rs`
**Effort:** 4 days
**Description:** Track object lifecycle states through CFG traversal. Define state machines for common resources (File, Mutex, DB connection). Flag violations: use-after-close, double-free, resource leaks, lock-without-unlock. Based on Dig 9 research.
**Acceptance criteria:**
- [ ] `TypeStateAnalyzer` walks CPG's CFG edges tracking abstract state per variable
- [ ] State machine definition format (Rust enum or data-driven config)
- [ ] File state machine: Closed -> Opened -> Read/Write -> Closed; detects use-after-close, leak
- [ ] Mutex state machine: Unlocked -> Locked -> Unlocked; detects double-lock, lock leak
- [ ] DB connection state machine: Closed -> Connected -> Queried -> Closed; detects query-without-connect, leak
- [ ] At merge points (if/else joins), take union of states (sound overapproximation)
- [ ] CWE-416 (use-after-free), CWE-675 (duplicate operations), CWE-404 (resource leak) in findings
- [ ] Minimum 20 tests covering Python file handling, Rust mutex, and DB connections
- [ ] False positive rate <10% on test fixtures

### Task 5.5 -- foundation crew: Coverage-informed severity re-scoring
**Files:** `crates/apex-coverage/src/oracle.rs`, `crates/apex-detect/src/finding.rs`, `crates/apex-detect/src/pipeline.rs`
**Effort:** 1 day
**Description:** Use the Compound Coverage Oracle (Task 3.4) to adjust finding severity. Findings in uncovered code get amplified (1.3x), findings in well-tested code get reduced (0.85x). Findings in production-reachable but untested code get maximum amplification (2.0x).
**Acceptance criteria:**
- [ ] `Finding.adjusted_severity` field computed from coverage data
- [ ] Multiplier table: 0% coverage = 1.3x, 1-50% = 1.1x, 51-90% = 1.0x, 91-100% = 0.85x
- [ ] Production-reachable + untested = 2.0x multiplier (when production trace data available)
- [ ] Severity adjustment is opt-in (enabled when coverage data present, no-op without it)
- [ ] Original severity always preserved in `Finding.base_severity`
- [ ] Test: finding in uncovered code gets amplified, finding in well-tested code gets reduced

## Verification Protocol

Every task follows TDD:
1. Write failing test
2. Run test, confirm failure
3. Implement
4. Run test, confirm pass
5. `cargo clippy --workspace -- -D warnings`
6. `cargo fmt --check`
7. Commit with descriptive message

Full workspace verification at wave boundaries:
```bash
cargo check --workspace
cargo nextest run --workspace
cargo clippy --workspace -- -D warnings
```
